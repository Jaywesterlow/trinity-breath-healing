<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → GitHub UI / gh CLI | Repo creation + branch protection requires authenticated user |
| user → Vercel dashboard | Project binding + env var management requires authenticated user |
| user → Google Search Console | Property registration requires Google account |
| public CDN → AI crawlers | Production URL is publicly indexable; preview URLs may be too |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-deploy | Tampering | Vercel env vars compromised | mitigate | (1) PUBLIC_SITE_URL is plaintext (public URL); no secrets in Phase 0 (Resend/Cal.com keys land in Phase 3 with proper secret discipline); (2) Vercel project-level env vars restrict to Production environment when secrets land — documented in PHASE0-LAUNCH-CHECKLIST.md as a forward-reference for Phase 3 |
| T-00-deploy | Spoofing | fork PR running with main-branch secrets | accept (Phase 0) / mitigate (Phase 3) | Repo is private; only the user can open PRs. Phase 3 will additionally require: deny fork PR access to Vercel preview deploys with secret env vars (Vercel Deployment Protection setting). |
| T-00-noindex-preview | Information Disclosure | Vercel preview URLs indexed by search engines | mitigate | Plan 02 shipped the `meta.noindex` plumbing; Vercel "Deployment Protection" can be enabled to add `X-Robots-Tag: noindex` to all preview deployments. Decide here whether to enable Vercel Deployment Protection OR to flip `PUBLIC_VERCEL_ENV !== 'production' → meta.noindex=true` in Plan 02's defaults — pick one approach in this checkpoint and document in PHASE0-LAUNCH-CHECKLIST.md. Recommended: enable Vercel Deployment Protection ("noindex" toggle on preview) — minimal code surface, vendor-managed. |
| T-00-htmlaudit | Tampering | required status checks bypass | mitigate | Branch protection rule (Task 3) makes the 3 CI jobs required; the user cannot merge with failures unless they explicitly remove the rule, which is a deliberate self-override and acceptable for solo dev |

</threat_model>
