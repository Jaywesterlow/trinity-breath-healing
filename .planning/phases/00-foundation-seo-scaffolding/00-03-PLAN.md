<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| schema source files → JsonLd component | TypeScript-typed via schema-dts; non-Schema.org @type values are compile errors |
| JsonLd `{@html}` emission | Svelte unsafe-HTML opt-in; payload is JSON.stringify of typed data, NOT user input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-jsonld | Tampering | <JsonLd> `{@html}` emission | mitigate | (1) Source data is `schema-dts`-typed — no untrusted strings enter the payload from path/query at this stage; (2) defense in depth: `.replace(/<\/script/gi, '<\\/script')` on the serialized JSON before emit (per RESEARCH Security Domain recommendation); (3) Test #2 verifies the escape works when malicious string is in graph data. If Phase 1 ever pulls JSON-LD content from CMS / user input, this mitigation must be revisited. |
| T-00-meta | Tampering | shared.ts/services.ts data sources | mitigate | Schema data comes from `src/lib/constants/brand.ts` (committed source) and `$env/dynamic/public.PUBLIC_SITE_URL` (Vercel-managed). No query string, no path-derived strings. |

</threat_model>
