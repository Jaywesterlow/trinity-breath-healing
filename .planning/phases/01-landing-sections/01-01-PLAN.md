Update Head.svelte: add optional heroPreload prop to the $props() destructuring:
  `let { meta, heroPreload }: { meta: PageMeta; heroPreload?: string } = $props();`

In the svelte:head block, after the existing Twitter Card meta tags, add:
```svelte
{#if heroPreload}
  <link rel="preload" as="image" fetchpriority="high" href={heroPreload} />
{/if}
```

This approach (Option A per RESEARCH.md Pattern 7) keeps preload logic centralized in Head.svelte. The heroPreload value for the landing page is the resolved URL of the hero image. Because @sveltejs/enhanced-img transforms imports at build time, the easiest approach is to pass the `src` attribute of the rendered EnhancedImage as the preload href. In practice for SvelteKit SSG, the ?enhanced import resolves to a hashed string. Pass HeroImg (the imported value) directly from +page.svelte. However, note that HeroImg from ?enhanced is an object with src property — inspect EnhancedImage.svelte to determine the type. If HeroImg is an EnhancedImageData object, use `HeroImg.img.src` or the correct property. Import HeroImg in +page.svelte and pass it as heroPreload.

ALTERNATIVE (simpler, equally valid per RESEARCH open question 1): Instead of prop-threading, add `<svelte:head>` inside HeroSection.svelte directly:
```svelte
<svelte:head>
  <link rel="preload" as="image" fetchpriority="high" href={HeroImg.img?.src ?? HeroImg} />
</svelte:head>
```
This does NOT require a browser guard — svelte:head is rendered server-side in SSR. Choose whichever approach works with the actual EnhancedImage import type. Verify by inspecting the built HTML for the preload link.
